Widget HTML Markup Specifications

Requirements for Code Generator Compatibility

Extension Overview Code Generator Markup Specs

Required HTML Structure

Your widget HTML must follow this specific structure for the code generator to work correctly:

Rule 1: Outer Container with Unique Class

The widget must have a single outer container element with a unique CSS class that identifies the widget type.

<div class="your-widget-name">
  <!-- widget content -->
</div>
Rule 2: Repeating Item Structure

If your widget contains multiple items (plans, members, testimonials, etc.), each item must have a unique identifier attribute following the pattern data-{itemtype}-id.

<div class="team-member-bio">
  <div data-member-id="1">...</div>
  <div data-member-id="2">...</div>
  <div data-member-id="3">...</div>
</div>
Rule 3: Data Attributes for All Editable Fields

Every piece of data you want to edit must be stored in a data-* attribute on the item element. The generator will detect these automatically.

<div data-member-id="1"
     data-name="John Smith"
     data-title="Senior Developer"
     data-email="john@company.com"
     data-phone="555-1234"
     data-bio="Biography text here">
  <!-- visible content -->
</div>

Naming Conventions

Widget Class Names

  • Use lowercase with hyphens (kebab-case)
  • Be descriptive and unique
  • Avoid generic names like "widget" or "component"

Good Examples

team-member-bio
insurance-comparison-tool
testimonial-carousel
pricing-calculator
product-showcase

Bad Examples

widget
component
box
container
content

Data Attribute Names

  • Use the pattern data-{field-name}
  • Use lowercase with hyphens
  • Be descriptive and clear
  • For item identifiers, use data-{itemtype}-id
Attribute Name Purpose Generated Form Field
data-member-id Unique identifier (required) Hidden (used for tracking)
data-name Person's full name Text input: "Name"
data-email Email address Email input: "Email"
data-phone Phone number Tel input: "Phone"
data-bio Biography text Textarea: "Bio"
data-profile-url URL to profile URL input: "Profile URL"

Visible Content Structure

Rule 4: CSS Classes Match Data Attributes

For the generated code to work with minimal customization, your visible content elements should use CSS classes that match the data attribute names.

Pattern: If you have data-{field-name}, use class .{itemtype}-{field-name}

Matching Pattern (Recommended)

<div data-member-id="1"
     data-name="John Smith"
     data-email="john@company.com">
  <h3 class="member-name">
    John Smith
  </h3>
  <p class="member-email">
    john@company.com
  </p>
</div>

Non-Matching (Requires Manual Updates)

<div data-member-id="1"
     data-name="John Smith"
     data-email="john@company.com">
  <h3 class="person-fullname">
    John Smith
  </h3>
  <p class="contact-email-address">
    john@company.com
  </p>
</div>
Note: If your CSS classes don't match the pattern, the generator will still create the code, but you'll need to manually update the querySelector selectors in the update function.

Pre-Generation Checklist

Before using the code generator, verify your HTML meets these requirements:

  1. ✓ Unique outer container class — One clear class name for the widget
  2. ✓ Item identifier pattern — data-{itemtype}-id on each repeating item
  3. ✓ All data in attributes — Every editable field has a data-* attribute
  4. ✓ Consistent naming — Lowercase, hyphenated attribute names
  5. ✓ Matching CSS classes — .{itemtype}-{field-name} pattern for visible elements
  6. ✓ Complete sample — Include at least 2 items showing all possible fields
  7. ✓ Valid HTML — Properly nested and closed tags

Troubleshooting

Generator says "No data-* attributes found"

  • Make sure you're using data- prefix (not just custom attributes)
  • Check that attributes are on the actual elements, not in comments
  • Verify the HTML is valid and can be parsed

Generator can't detect item type

  • Add a data-{itemtype}-id attribute to identify repeating items
  • Example: data-member-id, data-plan-id, data-product-id
  • Use singular form (member, not members)

Generated selectors don't match my HTML

  • Follow the .{itemtype}-{field-name} class naming pattern
  • OR manually update the selectors in the generated code
  • Look for lines like el.querySelector('.member-name') and adjust
