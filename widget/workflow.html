The Full Lifecycle

Five phases from zero to managed widget

New widgets follow a consistent five-phase lifecycle. Once set up, ongoing content updates skip all the way to Phase 5 — authors open the sidebar and edit, nothing else changes.

Phase 1

Build the Widget

Author the HTML, CSS, and JS as a standard self-contained widget. Assign a unique root CSS class — this is the only selector the extension needs to discover the widget.

  • Root class — unique, e.g. .animated-cards
  • Repeat items — consistent child selector
  • Field selectors — stable classes on editable elements
  • Data attributes — if values need syncing
Phase 2

Split HTML and JS

Separate the widget into two deployable pieces. This is required — not optional.

  • Sitecore RTE — CSS in a scoped <style> tag + all HTML markup
  • GTM Custom HTML — JavaScript only
  • GTM tag triggers on DOM Ready
  • No JS in the Sitecore field
Phase 3

Register the Widget

Add a widget definition to widget-registry.json on the server. No extension files change.

  • Add entry to widgets array
  • Specify root class and display name
  • Define fields with selectors and labels
  • Bump the registry version string
  • Push JSON to server
Phase 4

Deploy to Sitecore

Paste the Sitecore component HTML into a Rich Text field. Add the GTM tag to the appropriate container.

  • Paste HTML into Sitecore RTE via "Edit HTML"
  • Publish the item
  • Add GTM tag, trigger on affected pages
  • Verify widget renders correctly
  • Confirm extension detects the widget
Phase 5

Content Management

All ongoing content updates happen through the sidebar. No code changes needed for data edits.

  • Open Content Editor
  • Navigate to the component item
  • Open sidebar via extension icon
  • Edit fields, click Apply Changes
  • Save in Sitecore
Ongoing

Adding / Updating Widgets

Structural changes (new fields, new widgets) require only a registry edit. No extension rebuild, no reinstall for users.

  • Edit widget-registry.json on server
  • Bump the version string
  • Push — sidebar picks it up on next open
  • Extension icon badge notifies users of update
The HTML / GTM Split

Why every widget is two files

01

Sitecore Rich Text Component — HTML + CSS only

The component pasted into the Sitecore Rich Text field contains the scoped <style> block and all HTML markup. This is the version the extension reads and writes. It must contain no JavaScript.

  • Scoped <style class="[widget]-styles"> tag
  • Root element with the registry CSS class
  • All repeat items (.profile-card, .plan-button, etc.)
  • All editable elements with stable class selectors
  • Data attributes where values need syncing (e.g. data-target)
  • No inline <script> tags
  • No external JS imports
  • CSS scoped inside the root selector
⚠ If JS lives in the Sitecore field, GTM will re-inject it on every page load and overwrite the extension's DOM edits before the author can save. Keep JS in GTM only.
02

GTM Custom HTML Tag — JavaScript only

The GTM tag contains only the widget's behavior script. It reads the HTML structure from the DOM that Sitecore rendered — it does not inject or re-render HTML itself.

  • Wrapped in an IIFE (function () { ... })()
  • Trigger: DOM Ready
  • Scoped to pages where the widget appears
  • Reads from existing DOM elements by selector
  • No HTML template strings
  • No innerHTML overwrites of the widget root
  • IntersectionObserver for scroll-triggered effects
  • Guarded with existence checks before DOM queries
The GTM tag's job is to animate or enhance what's already in the DOM — never to recreate it.
Registry Updates

Adding or changing a widget — no extension rebuild

The registry is the only file that needs to change when adding a new widget or updating field definitions. The extension has no awareness of specific widgets until it fetches the registry at runtime.

01

Open widget-registry.json on the server

Hosted at tylervinson.com/widget/widget-registry.json. Edit directly or via your deployment workflow.

02

Add or modify the widget definition

Each widget entry needs a unique class (the CSS selector on the root element), a name for the sidebar UI, and an array of field definitions. Repeating items go in repeat + fields; top-level fields go in standaloneFields.

Minimum widget entry
{ "class": "my-widget", "name": "My Widget", "fields": [ ... ] }
03

Bump the version string

The top-level "version" in the registry is compared against the locally cached version. Any change — including minor edits — should bump this string so the update badge appears on the extension icon for users who already have it open.

Version bump example
"version": "2.3.5" → "2.3.6"
04

Push to server — done

No extension ZIP rebuild. No reinstall prompt for users. The next time any author opens the sidebar, the new registry is fetched automatically. If the version changed, the icon badge and in-sidebar banner appear.

Changes are live for all users the moment the file is deployed. There is no caching layer in the extension — widget-registry.json is always fetched fresh.
Day-to-Day Use

Content update workflow for authors

Once a widget is deployed, content authors follow a simple repeatable flow — no code, no WYSIWYG source editing required.

1

Open Sitecore Content Editor

Navigate to the Sitecore item that contains the widget in its Rich Text field. The Quick Info panel on the right shows the item ID — the extension reads this automatically.

2

Click the extension icon

The sidebar opens. If a registered widget is detected on the current item, the editor form populates immediately. If no widget is found, a "no widgets detected" message is shown.

3

Edit fields in the form

Standalone fields (headings, subheadings) appear at the top. Repeating items appear as collapsed groups labeled with the item name (e.g. "Plan 1 — Bronze"). Expand any group to edit its fields. Rich text fields open a Quill editor.

4

Click Apply Changes

The extension fetches the current field HTML via the SSC API, applies all edits using the registry selectors, then silently automates the HTML editor: opens it, writes the updated markup, and closes it. The entire cycle completes in under a second.

5

Save in Sitecore

Click Save in the Content Editor toolbar (or use Ctrl+S). The extension's bridge has already fired all four change-notification strategies, so the Save button is active and ready. The item is saved and can be published as normal.
