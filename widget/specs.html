How It Works

Request & Response Flow

What happens from the moment the sidebar opens to when Apply Changes is pressed.

Startup sequence

Every time the sidebar opens, it fetches the latest registry from the server before doing anything else.

1. Sidebar opens → fetches widget-registry.json from the server.

2. Registry loaded → sidepanel reads the current item ID from the Content Editor's input.scEditorHeaderQuickInfoInput (the Quick Info panel) and sends GET_WIDGETS to the content script.

3. Content script calls the Sitecore SSC API: /sitecore/api/ssc/aggregate/content/Items('{id}')/Fields. It scans every Rich Text field in the response for registered widget CSS classes and returns any matches along with the field's item ID and field ID.

4. If a widget is found, the sidepanel immediately sends GET_WIDGET_DATA — passing the full field definition plus the discovered item ID and field ID.

5. Content script fetches the field HTML again via SSC API, parses it with DOMParser, and reads each field value using the selectors in the definition.

6. Sidepanel builds the form from field definitions and populates it with the current values. No WYSIWYG modal is opened at any point.

Apply Changes

When the user clicks Apply, the sidepanel collects form values and sends them with the full field definition back to the content script.

The sidepanel collects form values and sends them with the full field definition and field ID to the content script via UPDATE_WIDGET_DATA.

The content script fetches the current field HTML via SSC API, parses it, iterates each field definition, and applies updates — using selector for inline elements, gridSelector/gridCellSelector for comparison table cells, and syncDataAttr for data attributes. It then serializes the modified HTML back to a string.

Finally it programmatically clicks the "Edit HTML" button (not "Show editor"), waits for the textarea to appear, writes the updated HTML string into it, triggers a change event, and clicks Accept. The entire open/write/close cycle completes in under a second with no visible modal interaction for the user.

No item IDs in the registry

Sitecore item IDs and field IDs are discovered at runtime — the extension reads the selected item ID from the Content Editor's Quick Info panel and fetches all fields from the SSC API. This means the registry only needs CSS class names and selectors. Moving a widget to a different item or field in Sitecore requires no registry change.

Registry Schema

widget-registry.json

The single file that defines every supported widget. Hosted at /widget/widget-registry.json.

Top-level structure

{
  "version": "1.3.0",        // checked against extension version for update badge
  "widgets": [ ... ]      // array of widget definitions
}

Widget definition

One object per widget in the widgets array.

{
  "class": "insurance-comparison-tool",  // CSS class on the widget root element
  "name": "Insurance Plan Comparison",    // display name in sidebar picker
  "repeat": { ... },                   // optional — defines repeating items
  "fields": [ ... ],                   // fields on each repeat item
  "standaloneFields": [ ... ]          // optional — fields on the widget root
}
Property Description
classRequiredCSS class on the widget's root element. Used to detect the widget on the page.
nameRequiredHuman-readable label shown in the sidebar widget picker when multiple widgets are on the page.
repeatOptionalDefines a repeating group of items (e.g. plan buttons, stat cards). If omitted, only standaloneFields are used.
fieldsOptionalArray of field definitions scoped to each repeat item. Required if repeat is defined.
standaloneFieldsOptionalArray of field definitions scoped to the widget root — headings, descriptions, or any non-repeating element.

repeat block

Defines the repeating item pattern for a widget with multiple cards, rows, or buttons.

"repeat": {
  "selector": ".plan-button",       // CSS selector for each item, relative to widget root
  "groupLabel": "Plan",             // label for each group in the sidebar (e.g. "Plan 1", "Plan 2")
  "groupLabelField": "name"         // optional — field key whose value is appended to the group header
}
PropertyDescription
selectorRequiredCSS selector used to find each repeated item within the widget root.
groupLabelRequiredWord used to label each group in the sidebar. Groups are numbered automatically: "Plan 1", "Plan 2", etc.
groupLabelFieldOptionalKey of a field in fields whose current value is appended to the group header. As the user edits the name field, the header updates live.
Field Definitions

Mapping Selectors to Form Fields

Every editable element in the widget is described by a field definition object.

Standard field

A field that reads and writes a single DOM element within each repeat item.

{
  "key": "name",                     // internal identifier, used as form field ID
  "label": "Plan Name",             // label shown above the input in the sidebar
  "selector": ".plan-button-name", // CSS selector relative to the repeat item
  "type": "text"                    // "text" (default) or "textarea"
}
PropertyDescription
keyRequiredUnique identifier for this field within the widget. Used internally as the form input ID and data key.
labelRequiredHuman-readable label shown above the form input in the sidebar.
selectorRequiredCSS selector for the target element, evaluated relative to the repeat item (or widget root for standalone fields).
typeOptionalInput type. text (single-line, default) or textarea (multi-line).

Data attribute sync

Fields that also need to update a data-* attribute — used for animated counters and GTM tracking.

{
  "key": "price",
  "label": "Monthly Price",
  "selector": ".plan-button-price",
  "syncDataAttr": "data-price",    // attribute name to set on the repeat item element
  "syncNumeric": true              // strips non-numeric characters before writing the attribute
}
syncNumeric: When true, the value written to the data attribute has all non-numeric characters stripped. For example, "$299/mo" becomes "299". Use this for animated counter targets and any attribute that expects a raw number.

Grid fields

Fields that update a value in a separate comparison table row in addition to (or instead of) the repeat item itself.

{
  "key": "deductible",
  "label": "Deductible",
  "gridSelector": "[data-comparison='deductible']", // selects the table row, relative to widget root
  "gridCellSelector": "[data-plan='{planNum}']"           // selects the cell within the row; {planNum} is replaced with 1-based item index
}

Grid-only fields have no selector — they write exclusively to the comparison table. Fields with both selector and gridSelector write to both locations when Apply Changes is pressed.

{planNum}: The placeholder {planNum} in gridCellSelector is replaced at runtime with the 1-based index of the current repeat item. Item 0 becomes plan 1, item 1 becomes plan 2, and so on.
Full Example

Annotated Widget Definition

A complete registry entry for the Insurance Plan Comparison widget.

{
  "class": "insurance-comparison-tool",
  "name": "Insurance Plan Comparison",

  // Standalone fields — scoped to the widget root, not to repeat items
  "standaloneFields": [
    { "key": "title",    "label": "Heading",    "selector": "h1" },
    { "key": "subtitle", "label": "Subheading", "selector": ".subtitle" },
    { "key": "instructionHeading", "label": "Instruction Heading", "selector": ".instruction-content h2" },
    { "key": "instructionBody",    "label": "Instruction Body",    "selector": ".instruction-content p", "type": "textarea" }
  ],

  // Repeat block — each .plan-button is one group in the sidebar
  "repeat": {
    "selector": ".plan-button",
    "groupLabel": "Plan",
    "groupLabelField": "name"   // group header shows live value of the "name" field
  },

  // Fields scoped to each repeat item (.plan-button)
  "fields": [

    // Writes to .plan-button-name AND syncs data-plan-name attribute
    {
      "key": "name", "label": "Plan Name",
      "selector": ".plan-button-name",
      "syncDataAttr": "data-plan-name"
    },

    // Writes to .plan-button-price, syncs data-price as numeric, AND updates comparison table cell
    {
      "key": "price", "label": "Price",
      "selector": ".plan-button-price",
      "syncDataAttr": "data-price", "syncNumeric": true,
      "gridSelector": "[data-comparison='price']",
      "gridCellSelector": "[data-plan='{planNum}']"
    },

    // Grid-only field — no .plan-button element, writes only to the comparison table
    {
      "key": "deductible", "label": "Deductible",
      "gridSelector": "[data-comparison='deductible']",
      "gridCellSelector": "[data-plan='{planNum}']"
    }

    // ... additional fields follow the same patterns
  ]
}
Adding a Widget

Step-by-Step

What a developer needs to do to make a new widget editable.

1. Identify the widget root class. Every widget needs a unique CSS class on its outermost container element. If the widget doesn't have one, add it.

2. Map the editable fields. For each element a content author needs to edit, note its CSS selector relative to the widget root (or relative to each repeating item, if applicable). Decide which fields need data attribute sync and which update a grid table.

3. Add the widget definition to the registry. Open widget-registry.json, add a new entry to the widgets array following the schema above. Increment the version string.

4. Deploy the registry. Push the updated widget-registry.json to the server at /widget/widget-registry.json. No changes to the extension are required.

5. Verify. Open the extension sidebar in the Content Editor and navigate to a component that contains the new widget. The form should populate automatically with the current field values. If fields are missing or blank, check that the CSS selectors resolve correctly against the HTML stored in the rich text field.

Version bump: Incrementing version in the registry causes the extension's background worker to set the update badge (!) for any user whose installed extension version doesn't match. This is the signal to content authors that a registry update is available.
