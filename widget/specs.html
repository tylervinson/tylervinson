How It Works

Request & response flow

What happens from the moment the sidebar opens to when Apply Changes is pressed.

Startup sequence

Every open fetches the latest registry before any Sitecore interaction.

1. Sidebar opens → fetches widget-registry.json from the server (no cache).

2. Registry loaded → sidepanel reads the current item ID from input.scEditorHeaderQuickInfoInput (the Content Editor's Quick Info panel) and sends GET_WIDGETS to the content script with the full registry.

3. Content script calls the SSC API: /sitecore/api/ssc/aggregate/content/Items('{id}')/Fields. It scans every Rich Text field in the response for registered widget CSS classes and returns any matches, including field ID and item ID.

4. If a widget is found, sidepanel sends GET_WIDGET_DATA with the full widget definition plus the discovered item and field IDs.

5. Content script fetches the field HTML via SSC API again, parses it with DOMParser, and reads each field value using the selectors from the definition.

6. Sidepanel builds the editor form from the field definitions and populates it with the current values. No WYSIWYG modal is opened at any point during read.

Apply Changes

The full write cycle — from form collection to Sitecore save activation.

1. Sidepanel collects all form values and sends them with the full field definition and field ID to the content script via UPDATE_WIDGET_DATA.

2. Content script fetches the current field HTML via SSC API and parses it with DOMParser.

3. It iterates all field definitions, applying updates: selector for standard elements, gridSelector/gridCellSelector for comparison table cells (with {planNum} substitution), and syncDataAttr to keep data attributes in sync with text content.

4. The modified DOM is serialized back to an HTML string.

5. The content script programmatically clicks the "Edit HTML" button in the WYSIWYG toolbar (not "Show editor"), waits for the textarea to appear, writes the updated HTML string into it, triggers a change event, and clicks Accept. The entire cycle completes in under a second.

6. sitecore-bridge.js fires all 4 change-notification strategies to activate the Sitecore Save button.

No item IDs or field IDs in the registry

Sitecore item IDs and field IDs are discovered at runtime from the Quick Info panel and SSC API. The registry only contains CSS class names and selectors. Moving a widget to a different Sitecore item or renaming a field requires no registry change.

Registry Schema

widget-registry.json

The single source of truth for all widgets. Hosted at tylervinson.com/widget/widget-registry.json. Currently at version 2.3.5.

Top-level structure

{
  "version":    "2.3.5",            // bump to trigger update badge in extension
  "updatedAt":  "2026-06-04T00:00:00.000Z",
  "downloadUrl": "https://www.tylervinson.com/widget/",
  "widgets":    [ ...widget definitions ]
}

Widget definition object

One entry per widget in the widgets array.

{
  "class":           "animated-cards",  // CSS class on the widget root element
  "name":            "Animated Cards",  // display name shown in sidebar
  "standaloneFields": [ ... ],           // optional — fields scoped to root
  "repeat":          { ... },            // optional — defines repeating items
  "fields":          [ ... ]             // fields inside each repeat item
}
PropertyRequiredDescription
classRequiredCSS class on the widget's root element. Used to detect the widget in Rich Text field HTML.
nameRequiredHuman-readable label shown in the sidebar widget picker when multiple widgets are found.
repeatOptionalDefines a repeating group of items (cards, plans, meetings). If omitted, only standaloneFields are used.
fieldsOptionalArray of field definitions scoped to each repeat item. Required when repeat is defined.
standaloneFieldsOptionalFields scoped to the widget root — headings, subheadings, or any single non-repeating element.

repeat block

Defines the repeating item pattern and how groups are labeled in the sidebar.

"repeat": {
  "selector":       ".profile-card",  // CSS selector for each item, relative to widget root
  "groupLabel":     "Card",           // label prefix — "Card 1", "Card 2", etc.
  "groupLabelField": "title",          // optional — appends the value of this field to the label
  "dateField":      "datetime",      // optional — field key used for date expiry checks
  "dateAttr":       "data-end"        // optional — data attribute holding the ISO end date
}
PropertyRequiredDescription
selectorRequiredCSS selector for each repeated item, evaluated relative to the widget root.
groupLabelRequiredWord used to prefix each group header. "Plan" produces "Plan 1", "Plan 2", etc.
groupLabelFieldOptionalKey of a field whose current value is appended to the group header. Updates live as the user types.
dateFieldOptionalField key for a datetime field — used to visually flag expired items in the editor.
dateAttrOptionalData attribute on the repeat item that holds the ISO end date string for expiry detection.

Field definition objects

Each editable element in the widget has a field definition in fields or standaloneFields.

// Standard text field
{ "key": "name", "label": "Plan Name", "selector": ".plan-button-name", "type": "text" }

// Textarea field
{ "key": "body", "label": "Description", "selector": ".meeting-body", "type": "textarea" }

// Rich text field (Quill editor)
{ "key": "detail", "label": "Detail Body", "selector": ".detail", "type": "richtext" }

// Field with data attribute sync
{ "key": "number", "label": "Statistic Value", "selector": ".number", "syncDataAttr": "data-target", "syncNumeric": true }

// Comparison grid cell (uses {planNum} substitution)
{ "key": "price", "label": "Monthly Price", "gridSelector": ".comparison-grid", "gridCellSelector": "[data-comparison='price'] [data-plan='{planNum}']" }
PropertyRequiredDescription
keyRequiredUnique identifier within the widget. Used as the form input ID.
labelRequiredHuman-readable label shown above the input in the sidebar.
selectorRequired*CSS selector for the target element, relative to the repeat item or widget root. Required unless using gridSelector.
typeOptionaltext (default), textarea, richtext (Quill editor), or datetime-local.
syncDataAttrOptionalIf specified, the data attribute on the target element is also updated to match the field value on Apply.
syncNumericOptionalBoolean. When true, the syncDataAttr value is written as a number (no trailing whitespace or symbols).
gridSelectorOptionalCSS selector for a comparison grid container. Used instead of selector for grid cell targeting.
gridCellSelectorOptionalSelector with {planNum} placeholder (1-based index). Resolved at runtime to target the correct column cell.
Registered Widget Definitions

All three current registry entries

Animated Cards

Scroll-triggered stat cards. Each card has a statistic value (synced to data-target for the JS counter) and a label. Simple repeat, no standalone fields.

{
  "class": "animated-cards",
  "name":  "Animated Cards",
  "repeat": {
    "selector":   ".profile-card",
    "groupLabel": "Card"
  },
  "fields": [
    {
      "key":          "number",
      "label":        "Statistic Value",
      "selector":     ".number",
      "type":          "text",
      "syncDataAttr": "data-target",  // keeps data-target in sync with .number text
      "syncNumeric":  true
    },
    {
      "key":      "text",
      "label":   "Card Label",
      "selector": ".copy-text",
      "type":    "text"
    }
  ]
}

Insurance Plan Comparison Tool

Side-by-side plan comparison with a button row and a comparison grid. Plan buttons are the repeat items; grid cells use {planNum} substitution for column targeting. Heading is a standalone field.

{
  "class": "insurance-comparison-tool",
  "name":  "Insurance Plan Comparison Tool",
  "standaloneFields": [
    { "key": "heading", "label": "Section Heading", "selector": "h1", "type": "text" }
  ],
  "repeat": {
    "selector":       ".plan-button",
    "groupLabel":     "Plan",
    "groupLabelField": "name"  // group header reads from the plan name field live
  },
  "fields": [
    { "key": "type",     "label": "Plan Type",     "selector": ".plan-button-type" },
    { "key": "name",     "label": "Plan Name",     "selector": ".plan-button-name" },
    { "key": "coverage", "label": "Coverage Type", "selector": ".plan-button-coverage" },
    // Grid fields — no selector; use gridSelector + gridCellSelector with {planNum}
    { "key": "price",        "label": "Monthly Price",       "gridSelector": ".comparison-grid", "gridCellSelector": "[data-comparison='price'] [data-plan='{planNum}']" },
    { "key": "deductible",   "label": "Annual Deductible",    "gridSelector": ".comparison-grid", "gridCellSelector": "[data-comparison='deductible'] [data-plan='{planNum}']" },
    // ...additional grid fields: maxOutOfPocket, coveragePercent, doctorVisits,
    //    prescription, emergency, mentalHealth, dentalVision
  ]
}

Zoom Meetings

Upcoming-sessions calendar. Each meeting is a repeat item with date/time, location, link, and rich text description. The dateAttr enables auto-expiry of past meetings. All internal elements use zmw- prefix to avoid collisions.

{
  "class": "zoom-meetings",
  "name":  "Zoom Meetings",
  "repeat": {
    "selector":        ".zmw-meeting",
    "groupLabel":      "Meeting",
    "groupLabelField": "title",
    "dateField":       "datetime",  // field key for date display/expiry
    "dateAttr":        "data-end"   // DOM attribute holding ISO end date
  },
  "fields": [
    { "key": "title",    "label": "Meeting Title",  "selector": ".zmw-title",    "type": "text" },
    { "key": "datetime", "label": "Date & Time",    "selector": ".zmw-datetime", "type": "datetime-local" },
    { "key": "location", "label": "Location",       "selector": ".zmw-location", "type": "text" },
    { "key": "link",     "label": "Zoom Link URL", "selector": ".zmw-link",     "type": "text" },
    { "key": "detail",   "label": "Detail Body",   "selector": ".zmw-detail",   "type": "richtext" }
  ]
}
Chrome Messaging

Message types — sidepanel ↔ content script

All communication between sidepanel.js and content-script.js uses Chrome's chrome.tabs.sendMessage with typed message objects.

typeDirectionPurpose
GET_WIDGETS→ contentScan all Rich Text fields on the current item via SSC API. Payload includes the full registry. Returns array of detected widget matches.
GET_WIDGET_DATA→ contentRead field values for a specific widget. Payload includes widget definition, item ID, and field ID. Returns a keyed data object.
UPDATE_WIDGET_DATA→ contentApply field changes. Payload includes updated values, widget definition, field ID. Automates the HTML editor write cycle.
GET_SC_USERNAME→ contentRead the logged-in Sitecore username from the Content Editor DOM for the dashboard header.
GET_LOCKED_ITEMS→ contentFetch all checked-out (locked) items via SSC API for the dashboard "Checked Out" panel.
GET_SC_ITEM_NAME→ contentRead the current item ID, name, and path from the Content Editor Quick Info panel (used to populate recents).
NAVIGATE_TO_ITEM→ contentNavigate the Content Editor to a specific item ID via scForm.invoke('item:load').
Extension Files

File roles — v4.4.6

FileRole
manifest.jsonMV3 manifest. Declares sidePanel, activeTab, storage, webNavigation permissions. Registers content script load order (sitecore-bridge.js first, then content-script.js).
background.jsService worker. Opens the side panel on icon click. Checks registry version every 4 hours. Sets gold extension badge when an update is detected.
sidepanel.htmlPanel shell. Hosts the dashboard view and widget editor view. Loads quill.js and sidepanel.js.
sidepanel.jsMain panel logic. Fetches registry, drives dashboard (search, recents, locked items), detects and renders widget editor forms, routes all messages to content script.
content-script.jsSitecore interface layer. Handles all SSC API calls, widget detection, field reads, HTML editor automation, and Sitecore navigation. Injected into all frames.
sitecore-bridge.jsInjected first in every frame. Fires 4 change-notification strategies on any DOM write. Also handles __sitecoreBridgeNavigate events for item navigation.
widget-registry.jsonServer-hosted only (not bundled in the ZIP). The sole source of widget definitions. Fetched fresh on every sidebar open.
quill.js / quill.snow.cssBundled Quill rich text editor. Used for richtext type fields in the widget editor form.
The extension ZIP does not include widget-registry.json. The registry is always served from the server and never cached in the extension bundle.
