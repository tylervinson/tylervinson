Getting Started

Installation

The extension is installed manually via Chrome developer mode. It does not require any changes to your Sitecore environment or widget markup — it reads field values entirely through CSS selectors defined in the registry.

  • 1
    Download the extension ZIP Download from tylervinson.com/widget/ and unzip to a folder on your machine.
  • 2
    Open Chrome Extensions Navigate to chrome://extensions and enable Developer Mode using the toggle in the top right.
  • 3
    Load unpacked Click "Load unpacked" and select the unzipped extension folder. The Sitecore Widget Editor icon will appear in your toolbar.
  • 4
    Open the sidebar Navigate to a page containing a registered widget and click the extension icon. The sidebar opens and detects widgets automatically.

The Registry File

The registry is a single JSON file hosted at tylervinson.com/widget/widget-registry.json. It is the only thing that needs to change when you add or modify a widget. The extension fetches it fresh every time the sidebar opens.

What the registry controls

  • Which CSS classes identify editable widgets
  • What the widget is called in the editor UI
  • Which DOM elements are fields and what they're labelled
  • How repeating items are grouped (cards, rows, plans)
  • Which fields sync numeric values back to data attributes
  • The version string that triggers the update badge

What the registry does not control

  • Widget HTML structure — no special markup required
  • CSS or JavaScript — the extension does not inject styles
  • Sitecore field bindings — changes apply to the live DOM only
  • Field validation — values are written as-is
  • Field ordering in forms — follows array order in the registry
widget-registry.json — top-level structure 
{
  "version": "1.3.0",          // bump this to trigger the update badge
  "widgets": [ ... ]       // array of widget definition objects
}
Registry Schema

Widget Object

Each entry in the widgets array describes one editable widget. The extension identifies widgets on the page by matching class against elements in the DOM.

Property Type Description
class string Required The CSS class name on the widget's root element. The extension queries document.querySelectorAll('.' + class) to find instances on the page.
name string Required Display name shown in the sidebar picker when multiple widgets are detected on the same page.
repeat object Optional Defines a repeating group of items (cards, plans, rows). See the Repeat Block reference below. Omit for widgets with only standalone fields.
fields array Optional Field definitions scoped to each repeat item. Required if repeat is present. See the Field Object reference below.
standaloneFields array Optional Fields that appear once in the widget, outside any repeating item — headings, subtitles, introductory copy. Uses the same field definition format but selectors are relative to the widget root.

Repeat Block

The repeat object tells the extension how to find the repeating elements within a widget — the individual cards, plan buttons, or table rows that each have their own set of editable fields. The sidebar renders one collapsible group per item found.

Property Type Description
selector string Required CSS selector for each repeating item, relative to the widget root. E.g. .profile-card or .plan-button.
groupLabel string Required Singular label for each group in the sidebar. E.g. "Card" or "Plan". Groups are labelled "Card 1", "Card 2", etc.
groupLabelField string Optional The key of the field whose value is appended to the group header for context. E.g. if set to "name", groups display as "Plan 1 — Basic" using the current plan name value.

Field Object

Fields in the fields array are scoped to each repeat item. The selector property is resolved relative to the item element — not the widget root. A field can target the item element itself, a comparison table cell, or both simultaneously using selector and gridSelector together.

Property Type Description
key string Required Internal identifier for this field. Used to wire form inputs to DOM elements. Must be unique within the fields array for a given widget.
label string Required Human-readable label displayed above the input in the sidebar editor form.
selector string Optional* CSS selector for the target element, relative to the repeat item. Required unless gridSelector is the only target. Text content is read from and written to this element.
type string Optional Input type: "text" (default) or "textarea" for longer content. Affects the form input rendered in the sidebar.
syncDataAttr string Optional If set, the extension also writes the field value to this data-* attribute on the repeat item element. E.g. "data-target" keeps counter animation start values in sync.
syncNumeric boolean Optional When true, strips non-numeric characters before writing to syncDataAttr. Useful when the visible value includes a symbol (e.g. $299) but the data attribute expects a raw number (299).
gridSelector string Optional CSS selector for the corresponding row in a comparison table, relative to the widget root. When present, the field value is also written to the matching grid cell. Used when a value appears in both a card button and a table row.
gridCellSelector string Optional* Selector for the specific cell within the grid row, relative to the row. Use {planNum} as a placeholder — it is replaced with the item's 1-based index. E.g. "[data-plan='{planNum}']". Required if gridSelector is set.

Standalone Fields

The standaloneFields array works identically to fields except that selectors are resolved relative to the widget root rather than a repeat item. Use it for elements that appear once in the widget regardless of how many items it contains — a section heading, a subtitle, or an introductory paragraph.

Standalone fields are rendered above the repeating item groups in the sidebar, under a "General" heading. They support the same key, label, selector, and type properties. The syncDataAttr, syncNumeric, gridSelector, and gridCellSelector properties are not applicable to standalone fields.

standaloneFields example 
"standaloneFields": [
  {
    "key": "title",
    "label": "Section Heading",
    "selector": "h1",
    "type": "text"
  },
  {
    "key": "subtitle",
    "label": "Subtitle",
    "selector": ".subtitle",
    "type": "text"
  },
  {
    "key": "body",
    "label": "Intro Copy",
    "selector": ".intro-text p",
    "type": "textarea"
  }
]

Full Example

A complete registry entry for a widget with both standalone fields and repeating items. This example represents an animated stats card row with a section heading and four individual stat cards.

widget-registry.json 
{
  "version": "1.3.0",
  "widgets": [
    {
      "class": "card-row",
      "name": "Animated Stats Cards",
      "standaloneFields": [
        {
          "key": "heading",
          "label": "Section Heading",
          "selector": ".section-heading",
          "type": "text"
        }
      ],
      "repeat": {
        "selector": ".profile-card",
        "groupLabel": "Card"
      },
      "fields": [
        {
          "key": "number",
          "label": "Statistic Value",
          "selector": ".number",
          "type": "text",
          "syncDataAttr": "data-target",
          "syncNumeric": true
        },
        {
          "key": "label",
          "label": "Card Label",
          "selector": ".copy-text",
          "type": "text"
        }
      ]
    }
  ]
}

How It Works

Understanding the data flow helps when debugging why a field isn't reading or writing correctly.

Reading field values (on sidebar open)

  • Sidepanel fetches the registry from the server
  • Sends GET_WIDGETS with full registry to the content script
  • Content script queries DOM for each widget class, returns matches
  • Sidepanel sends GET_WIDGET_DATA with the widget definition
  • Content script walks field selectors, returns text content
  • Sidepanel renders the form with current values pre-filled

Writing field values (on Apply)

  • Sidepanel collects all form values and sends UPDATE_WIDGET_DATA
  • Content script sets textContent on each target element
  • If syncDataAttr is set, writes value to the data attribute too
  • If syncNumeric, strips non-numeric characters first
  • If gridSelector is set, updates the comparison table cell
  • Calls SitecoreBridge.notifyChange(el) after each write

Update Flow

The extension has a lightweight update notification system. It does not auto-update itself — it notifies content authors that a new version is available and provides a direct download link.

  • 1
    Developer updates the registry New widget added, field changed, or structural update made. The top-level version string is incremented (e.g. 1.3.0 → 1.4.0) and the JSON is pushed to the server.
  • 2
    Background worker detects the version bump The background service worker polls the registry URL periodically and compares the fetched version against CURRENT_VERSION in the extension code. If they differ, it sets the ! badge on the extension icon.
  • 3
    Update banner appears in the sidebar When the content author next opens the sidebar, the background worker sends an UPDATE_AVAILABLE message and a banner appears at the top of the panel with the new version number and a download link.
  • 4
    Content author downloads and reinstalls The author downloads the new ZIP, removes the old extension from Chrome, and loads the new one unpacked. Registry-only changes (new fields, adjusted selectors) do not require a reinstall — only changes to the extension code itself do.

When you need to reinstall vs. just push JSON

Push JSON only: adding a new widget, adding or removing fields, changing selectors, updating labels, or any change that only touches the registry definition.

Reinstall required: changes to the extension code itself — sidepanel.js, content-script.js, background.js, sitecore-bridge.js, or manifest.json. In this case, increment both the registry version and the CURRENT_VERSION constant in background.js.
