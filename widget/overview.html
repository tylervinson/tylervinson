What This Is

Visual widget editing inside the Content Editor

Data-driven Sitecore widgets store all their content — stats, plan prices, card labels, meeting details — directly in HTML markup inside Rich Text fields. Updating them normally means navigating raw source code in the WYSIWYG editor.

This extension eliminates that entirely. It opens as a persistent sidebar panel inside Sitecore's Content Editor, reads field values via the SSC REST API, and presents a clean structured form. Hit Apply Changes and the extension silently opens the HTML editor, writes the updated markup, and closes it. No manual WYSIWYG interaction required.

Widget support is defined in a server-hosted widget-registry.json fetched fresh on every open. Adding a new widget is a single JSON edit — no extension rebuild, no user reinstall.

Core Capabilities

What the extension does

Dashboard Home View

On open, the sidebar shows a dashboard with a Sitecore content search, recently viewed items, and checked-out (locked) items — all in one place.

Auto Widget Detection

The sidebar reads the selected item ID from the Quick Info panel, queries the SSC API for all Rich Text fields, and detects any registered widget CSS class automatically.

Registry-Driven Forms

The editor form is built entirely from the registry JSON — labels, field types, repeat groups, and standalone fields. Zero hardcoded logic in the extension code.

Grouped Repeat Items

Repeating items — plan buttons, stat cards, meeting entries — are displayed as collapsible labeled groups. Standalone fields like headings appear above the groups.

Rich Text Field Support

Fields marked richtext in the registry render a full Quill editor — supporting bold, italic, links, and lists — rather than a plain text input.

Silent WYSIWYG Writes

On Apply, the extension clicks "Edit HTML", writes updated markup into the textarea, triggers a change event, then clicks Accept — all invisible to the user in under a second.

Sitecore Change Notification

After writing, sitecore-bridge.js fires 4 strategies simultaneously — DOM events, TinyMCE dirty flag, PageModes ChromeManager, and data attribute toggle — to reliably activate the Save button.

Sitecore Content Search

The dashboard's search bar queries Sitecore's search API and returns matching content items — clickable to navigate and load directly in the Content Editor.

Registry Update Notifications

The background worker checks the registry version every 4 hours. When a new version is detected, a gold badge appears on the extension icon and a download banner appears in the sidebar.

Architecture

How the pieces fit together

The extension is split across five files with distinct roles. No widget definitions live in the extension itself — every widget is defined in the server-hosted registry.

widget-registry.json

Hosted at tylervinson.com/widget/. Defines every widget — class name, field selectors, labels, repeat structure, data attribute sync rules, date expiry. Fetched fresh on every sidebar open. Bump the version to trigger the update badge.

sidepanel.js

Fetches the registry. Drives the dashboard and widget editor views. Reads the current item ID from Quick Info, routes all Sitecore communication through the content script via Chrome messages.

content-script.js

Handles all direct Sitecore interaction. Calls the SSC API to detect and read widgets. On save, automates the HTML editor open → write → close cycle. Also powers content search, locked items, and navigation.

sitecore-bridge.js

Injected first. Fires 4 change-notification strategies after any DOM write to reliably activate Sitecore's Save button across different editor versions and configurations.

background.js

MV3 service worker. Opens the side panel on icon click. Checks for registry version updates every 4 hours. Sets the extension badge when a newer registry is detected.

SSC REST API

Sitecore's aggregate content API at /sitecore/api/ssc/aggregate/content/Items('{id}')/Fields. Used to read all Rich Text fields for the current item — no editor modal required.

Deployment Model

HTML in Sitecore. JS in GTM. Always separate.

Every widget is split into two deployable pieces. This is a deliberate architectural constraint — not a recommendation.

Sitecore Rich Text field — contains all CSS (inside a scoped <style> tag) and the HTML markup with all data attributes. This is the piece the extension reads and writes.

GTM Custom HTML tag — contains the widget's JavaScript only. Triggers on DOM Ready, scoped to the pages where the widget appears.

The reason for this split: when all three (HTML, CSS, JS) live in GTM, the tag re-injects them on every page load. Any DOM edits made by the extension are overwritten before the author can save. Separating them means the extension's writes to the Sitecore field are stable.
